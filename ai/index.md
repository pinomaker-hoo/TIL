# AI (인공지능)

## 1. AI / 머신러닝 / 딥러닝 / LLM의 관계

용어가 비슷해서 헷갈리기 쉽다. 포함 관계로 보면 다음과 같다.

```
┌─────────────────────────────────────────────────┐
│  AI (인공지능)                                   │
│  ┌───────────────────────────────────────────┐  │
│  │  Machine Learning (머신러닝)              │  │
│  │  ┌─────────────────────────────────────┐  │  │
│  │  │  Deep Learning (딥러닝)             │  │  │
│  │  │  ┌───────────────────────────────┐  │  │  │
│  │  │  │  LLM (대규모 언어 모델)       │  │  │  │
│  │  │  │  예: GPT, Claude, Gemini      │  │  │  │
│  │  │  └───────────────────────────────┘  │  │  │
│  │  └─────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

| 용어 | 설명 |
| --- | --- |
| AI | 인간의 지능적인 행동을 모방하는 모든 기술의 총칭 |
| 머신러닝 | 데이터로부터 패턴을 학습하는 AI의 한 분야 |
| 딥러닝 | 신경망(Neural Network)을 깊게 쌓아 학습하는 머신러닝 기법 |
| LLM | 대량의 텍스트로 학습된 거대 언어 모델. 우리가 흔히 쓰는 ChatGPT 같은 것 |

<br />

## 2. LLM이란 무엇인가

LLM(Large Language Model)은 **다음에 올 단어를 확률적으로 예측하는 모델**이다. 인터넷의 방대한 텍스트로 학습되어 있어, 질문에 그럴듯한 답을 만들어낸다.

> LLM은 본질적으로 "다음 단어 예측기"이다. 똑똑해 보이지만, 사실은 통계적으로 가장 그럴듯한 단어를 이어붙이고 있을 뿐이다.

대표적인 LLM:

| 모델 | 제공사 | 특징 |
| --- | --- | --- |
| GPT-4 / GPT-5 | OpenAI | 가장 널리 쓰이는 상용 모델 |
| Claude | Anthropic | 긴 문서 처리에 강함 |
| Gemini | Google | 멀티모달(이미지/영상) 강점 |
| Llama | Meta | 오픈소스 (직접 호스팅 가능) |
| Qwen | Alibaba | 오픈소스, 중국어/한국어 강점 |

<br />

## 3. 알아야 할 핵심 용어 5가지

### (1) 토큰 (Token)

LLM이 텍스트를 다루는 최소 단위이다. 단어 한 개일 수도 있고, 단어의 일부일 수도 있다.

```
"안녕하세요" → ["안녕", "하세요"]    (2개 토큰)
"Hello world" → ["Hello", " world"]  (2개 토큰)
```

**왜 알아야 하는가?** LLM API는 토큰 수로 과금된다. 입력/출력 모두 토큰으로 계산된다.

<br />

### (2) 컨텍스트 윈도우 (Context Window)

LLM이 한 번에 읽고 기억할 수 있는 **최대 토큰 수**이다. 이 한계를 넘으면 앞부분을 잊어버린다.

| 모델 | 컨텍스트 윈도우 |
| --- | --- |
| GPT-4 | 128,000 토큰 (약 책 1권) |
| Claude Opus 4.7 | 1,000,000 토큰 (약 책 8권) |
| Gemini 2.5 | 1,000,000+ 토큰 |

**왜 알아야 하는가?** 긴 문서를 통째로 넣을 수 없을 때 RAG가 필요해진다.

<br />

### (3) 임베딩 (Embedding)

텍스트를 **숫자로 이루어진 벡터(좌표)** 로 변환한 것이다. 의미가 비슷한 텍스트는 좌표상 가까운 위치에 놓인다.

```
"강아지"  → [0.21, -0.45, 0.88, ...]   (1,536차원 벡터)
"개"      → [0.22, -0.43, 0.87, ...]   (강아지와 매우 가까움)
"자동차"  → [-0.91, 0.32, 0.05, ...]   (강아지와 멀리 떨어짐)
```

**왜 알아야 하는가?** RAG의 핵심 기술이다. "비슷한 의미의 문서 찾기"가 좌표상 거리 계산으로 바뀐다.

<br />

### (4) 벡터 DB (Vector Database)

임베딩(벡터)을 저장하고, 주어진 벡터와 **가장 가까운 벡터들을 빠르게 찾아주는** 데이터베이스이다.

| 벡터 DB | 특징 |
| --- | --- |
| Pinecone | 매니지드 서비스, 사용 쉬움 |
| Chroma | 오픈소스, 로컬 개발에 적합 |
| Weaviate | 오픈소스, 자체 호스팅 가능 |
| pgvector | PostgreSQL 확장. 기존 DB에 그대로 추가 가능 |

**왜 알아야 하는가?** RAG에서 "검색" 역할을 담당한다.

<br />

### (5) 프롬프트 (Prompt)

LLM에게 주는 **입력 문장 전체**이다. 시스템 프롬프트(역할 지정) + 사용자 메시지로 구성되는 것이 보통이다.

```
[시스템 프롬프트]
너는 친절한 고객지원 봇이다. 한국어로 존댓말을 사용한다.

[사용자 메시지]
배송이 언제 오나요?
```

**왜 알아야 하는가?** RAG에서는 "검색 결과"를 프롬프트에 끼워넣는다.

<br />

## 4. LLM의 한계 — RAG/파인튜닝이 필요한 이유

LLM은 강력하지만 다음 세 가지 한계가 있다. 이 한계를 보완하기 위해 RAG와 파인튜닝이 등장한다.

### (1) 지식 컷오프 (Knowledge Cutoff)

LLM은 **학습이 끝난 시점까지의 지식만** 알고 있다. 어제 일어난 뉴스나 사내 위키 내용은 모른다.

```
"우리 회사의 휴가 정책 알려줘"
→ LLM: 학습 데이터에 없음. 답을 모름.
```

<br />

### (2) 환각 (Hallucination)

모르는 것도 **그럴듯하게 지어내는** 현상이다. LLM은 본질이 "다음 단어 예측기"라 사실 검증 능력이 없다.

```
"신라 제50대 왕은 누구인가?"
→ LLM: "신라 제50대 왕은 ○○왕이다." (실제로는 존재하지 않을 수 있음)
```

<br />

### (3) 도메인 특화 지식/말투 부족

법률, 의료, 사내 전문 용어 같은 **특정 도메인의 깊은 지식이나 일관된 말투**는 약하다.

<br />

## 5. 해결책 — RAG와 파인튜닝

두 가지 접근 방식이 있다. 한 줄 비교는 다음과 같다.

| 방법 | 한 줄 요약 | 비유 |
| --- | --- | --- |
| **RAG** | 외부 자료를 **검색**해서 프롬프트에 끼워넣음 | 오픈북 시험 |
| **파인튜닝** | 모델 자체를 **추가 학습**시킴 | 전문의 추가 교육 |
