# RAG vs 파인튜닝 — 언제 무엇을 써야 하는가

## 개요

실무에서 가장 자주 받는 질문 중 하나가 "RAG를 써야 하나요, 파인튜닝을 써야 하나요?"이다. 두 방법은 **목적 자체가 다르기 때문에** 경쟁 관계가 아니라 보완 관계에 가깝다. 이 문서는 둘 중 무엇을 언제 선택해야 하는지 실용적인 기준을 제시한다.

> 각 기술의 상세 내용은 [RAG 문서](./rag.md)와 [파인튜닝 문서](./fine-tuning.md)를 참고한다.

<br />

---

## 1. 한 줄 요약

> **RAG는 "지식"을 주입하고, 파인튜닝은 "능력/말투"를 학습시킨다.**

| 방법 | 비유 | 핵심 |
| --- | --- | --- |
| RAG | 오픈북 시험 | 답할 때 외부 자료를 본다 |
| 파인튜닝 | 전문의 추가 교육 | 답하는 방식 자체를 바꾼다 |

<br />

## 2. 핵심 비교표

| 항목 | RAG | 파인튜닝 |
| --- | --- | --- |
| 목적 | 외부 지식 주입 | 모델의 행동/형식 학습 |
| 데이터 갱신 | 쉬움 (DB만 업데이트) | 어려움 (재학습 필요) |
| 초기 구축 비용 | 낮음 | 높음 (GPU + 데이터 정제) |
| 운영 비용 | 보통 (벡터 DB 유지) | 낮음 (API 호출만) |
| 출처 제시 | 가능 | 불가능 |
| 환각 감소 | 효과적 | 부분적 |
| 응답 속도 | 느림 (검색 단계 추가) | 빠름 |
| 도메인 말투 학습 | 약함 | 강함 |
| 일관된 출력 형식 | 약함 | 강함 |
| 적은 데이터로 가능 | 가능 (문서만 있으면 됨) | 어려움 (수백~수천 건 필요) |
| 최신 정보 반영 | 즉시 가능 | 재학습 필요 |

## 3. 의사결정 플로우차트

```
            "무엇을 해결하려고 하는가?"
                      │
       ┌──────────────┼──────────────┐
       │              │              │
       ▼              ▼              ▼
 최신/사내 정보를   특정 말투/형식을  단순 작업
 답하게 하고 싶다    일관되게 만들고  (분류, 변환,
       │           싶다              간단한 응답)
       │              │              │
       ▼              ▼              ▼
     ─────         ─────          ─────────
     RAG          파인튜닝         프롬프트
                                   엔지니어링
                                  (먼저 시도)

           둘 다 필요하다 → RAG + 파인튜닝 (하이브리드)
```

## 4. 상황별 선택 가이드

### RAG가 적합한 경우

- "우리 회사 휴가 정책 알려줘" 같은 **사내 지식 Q&A**
- 매뉴얼/FAQ 기반 **고객지원 봇**
- 법령, 논문, 기술 문서 기반 **전문 검색 어시스턴트**
- 매일 업데이트되는 뉴스/공지/제품 정보 활용
- 답변에 **출처가 반드시 필요한** 시스템

### 파인튜닝이 적합한 경우

- 사용자 입력을 항상 **정해진 JSON 스키마**로 변환
- 우리 회사 **고유한 말투/톤** 학습 (예: "쿨하고 친근한 CS 톤")
- **분류 작업** (스팸/정상, 긍정/부정, 카테고리 N개)
- 특정 형식의 **요약 생성** (보고서 양식, 회의록 양식)
- 프롬프트가 너무 길어져서 **토큰 비용을 줄이고 싶을 때**

### 프롬프트 엔지니어링만으로 충분한 경우

- 일회성 작업, 빈도가 낮은 작업
- 데이터가 10건 미만
- 일반 상식 수준의 질문
- 빠르게 프로토타이핑하고 효과를 검증해야 할 때

## 5. 자주 하는 오해

### "파인튜닝하면 새 지식이 모델에 주입된다"

→ **부정확하다.** 파인튜닝은 사실 지식을 안정적으로 주입하기에 적합하지 않다. 잘못 학습되면 오히려 환각이 늘 수 있다. **지식 주입은 RAG가 훨씬 효과적이다.**

<br />

### "RAG는 옛날 기술이고 파인튜닝이 더 발전된 것이다"

→ **아니다.** 둘은 **서로 다른 문제를 해결**하는 기술이다. 실무 사례 중 다수는 RAG로 해결되며, 최신 빅테크 제품도 대부분 RAG 기반이다.

<br />

### "파인튜닝하면 모델이 더 똑똑해진다"

→ **부정확하다.** 파인튜닝은 모델을 **특정 작업에 특화**시키는 것이다. 오히려 일반 능력은 떨어질 수 있다(Catastrophic Forgetting).

<br />

### "RAG는 데이터만 많이 넣으면 잘 동작한다"

→ **아니다.** 청킹 전략, 임베딩 모델, 검색 정확도 튜닝이 매우 중요하다. 데이터 양보다 **검색 품질**이 결과를 좌우한다.

<br />

## 6. 실무 권장 순서

새 AI 기능을 만들 때는 다음 순서로 접근한다. **항상 쉬운 것부터 시도하고, 안 되면 다음 단계로 올라간다.**

```
  1단계: 프롬프트 엔지니어링
        ├─ 잘 동작하면? → 끝
        └─ 안 되면? → 다음 단계

  2단계: RAG 도입
        ├─ 외부/사내 지식이 필요한가?
        ├─ 잘 동작하면? → 끝
        └─ 말투/형식 문제가 남으면? → 다음 단계

  3단계: 파인튜닝 검토
        ├─ 학습 데이터 충분히 있는가? (수백 건 이상)
        ├─ 형식/말투 일관성이 핵심인가?
        └─ Yes → 파인튜닝 + (필요하면) RAG 병행
```

> **함정**: 처음부터 파인튜닝을 시도하는 것은 비추천이다. 비용이 크고, 사실 RAG나 프롬프트로 해결되는 경우가 대부분이다.

## 7. 하이브리드 — 둘 다 쓰기

실제 상용 서비스에서는 두 가지를 **함께** 쓰는 경우가 많다.

```
[파인튜닝]                    [RAG]
모델에 우리 말투,      +     매번 검색으로
JSON 출력 형식 학습          최신 정보 주입
        │                          │
        └──────────┬───────────────┘
                   ▼
            완성된 챗봇
       (우리 말투 + 최신 정보)
```

### 예시: 사내 고객지원 봇

- **파인튜닝**: 회사 고유의 친절한 말투, 답변 끝에 항상 "도움이 더 필요하시면 알려주세요" 추가
- **RAG**: 최신 제품 매뉴얼, FAQ에서 정보 검색

이렇게 결합하면 두 방식의 장점을 모두 얻을 수 있다.

<br />

## 8. 요약

- RAG = **지식 주입**, 파인튜닝 = **행동/형식 학습**
- 비용/난이도: 프롬프트 < RAG < 파인튜닝 순서로 시도
- 자주 바뀌는 정보는 무조건 RAG
- 일관된 형식/말투가 필요하면 파인튜닝
- 실무에서는 **하이브리드**가 흔하다
- 처음부터 파인튜닝하지 말 것. 먼저 프롬프트와 RAG로 검증한다
