# 3강. 식당 웨이터 비유로 통찰하는 비동기 논블로킹 아키텍처

## 개요

NodeJS가 Single Thread로 수만명의 동시 접속을 처리 할 수 있는 지에 대한 내용 정리

- 1만명 이상의 클라이언트가 동시에 접속할 때 서버가 버티지 못하는 현상을 C10K 문제라고 한다. 트래픽이 늘어남에 따라 스케일 업이나 스케일 아웃과 같은 물리적 서버 조치에 의존했지만 NodeJS는 OS 레벨의 I/O 다중화 기술을 활용하여 해결한다.

## 멀티 스레드와 동기식 블로킹 (Synchronous Blocking)의 한계

전통적인 서버 아키텍쳐에서는 사용자 요청 1개에 1개의 스레드를 매칭하는 방식을 사용합니다. 해당 방식을 사용하게되면 데이터베이스 조회등의 작업이 완료할 때까지 스레드는 대기하는 데, 이를 Blocking이라고 합니다. 요청이 동시에 1,000개가 들어오게 되면 스레드도 1,000개가 필요하고, 1개의 스레드가 생성될 때 마다 독립적인 메모리 공간(Thread Stack)이 할당되기에 Ram 고갈로 인한 Out of Memory로 서버가 다운될 수 있는 문제가 있습니다.

- CPU는 한 번에 하나의 작업만 가능하지만 여러개를 동시에 처리하는 것처럼 보이는 이유는 CPU가 스레드를 번갈아가며 실행하기 때문인데, CPU가 하던 일을 멈추고 다른 스레드의 작업 상태를 불러오는 것을 Context Switching이라고 한다. 스레드가 많아질수록 Context Switching을 하는 데 자원을 많이 사용하여 극도로 느려지는 Thrashing이 발생하기도 한다.

## 싱글 스레드와 비동기 논블로킹 (Asynchronous Non-blocking)의 우아함

NodeJS는 1개의 스레드로 전체를 커버한다. 주방으로 비유하자면, 웨이터(스레드)는 주문(요청)을 주방에 넘긴 뒤에 요리를 기다리지 않고(Non-blocking) 즉시 다른 요청을 받는 것으로 볼 수 있다. 주방에서 요리가 완성되면 큐에 음식(이벤트)를 올려두고 종을 치고, 웨이터는 이 큐를 확인하여 완성된 순서대로 처리합니다.

- Single Thread라고 해서 정말 스레드를 한 개를 사용하는 것이 아니라 File I/O, Network 통신과 같은 무거운 작업은 운영체제의 커널이 제공하는 고성능 이벤트 통지(Linux의 epoll, Mac의 kqueue등)을 통해 운영체제에 위임합니다. 즉 NodeJS의 메인 스레드는 교통 정리를 빠르게 하고 무거운 작업은 OS 커널에서 처리하는 겁니다.

## CPU 집약적 작업(CPU-bound Tasks)의 정의와 종류

비동기 아키텍쳐에서 제일 중요한 것은 어떠한 것이 CPU를 오래 사용하는가이다. 작업의 완료시간은 네트워크나 디스크의 속도가 아니라 CPU의 연산속도(클록 스피드)에 따라 결정되는 작업을 의미하는 데 외부의 응답을 기다리는 대기 시간이 거의 없고 CPU가 계산을 수행해야하는 일을 말한다.

예시로는 아래가 있다.

- 암호화 및 해싱: 비밀번호를 bcrypt로 암호화하거나, 대용량 데이터를 복호화하는 과정.
- 이미지 및 동영상 처리: 사진 크기를 조절(Resizing)하거나, 동영상 코덱을 변환(Transcoding)하는 연산.
- 대규모 데이터 파싱: 수십만 줄의 JSON 데이터를 객체로 변환하거나(JSON.parse), XML 등을 파싱하는 작업.
- 압축 및 해제: 대용량 파일을 ZIP이나 Gzip으로 압축하는 과정.
- 복잡한 수학적 계산: 인공지능 모델의 추론(Inference), 대규모 배열의 정렬, 복잡한 통계 수치 산출 등.

NodeJS에서 위의 작업을 하게되면, 웨이터가 주방에 가서 양파를 써는 동안 다른 주문을 받지 못 하는 것과 같다.

## 치명적 안티 패턴: 이벤트 루프 블로킹 (양파 썰기)

비동기 환경에서 가장 경계해야하는 건은 메인 스레드가 CPU 집약적 작업을 수행하는 것이다. 이러한 작업을 하게 될 경우 스레드가 그 작업을 수행하느라 이벤트 루프 전체가 멈추고 모든 요청이 정지되게 된다.

- 현업에서는 worker_threads 모듈을 사용해 libuv의 Thread Pool과는 완전히 분리된 별도의 실행환경(worker)를 띄우거나 메인 서버는 API Gateway 역할만 수행하고, 무거운 연산은 타 MS에게 넘기는 설계를 합니다.

## 코드 레벨에서의 동작 원리와 콜백(Callback)의 본질

```javascript
// [안티 패턴] 동기(Synchronous) 블로킹 방식
// 웨이터가 주방 앞에서 요리가 나올 때까지 멈춰버립니다.
const data = db.querySync("SELECT * FROM users");
console.log("데이터 조회가 끝났습니다.", data); // DB 조회가 끝나기 전까지 절대 실행되지 않음

// [권장 패턴] 비동기(Asynchronous) 논블로킹 방식
// 웨이터가 주방에 주문과 함께 '진동벨(콜백 함수)'을 남기고 즉시 다음 일로 넘어갑니다.
db.query("SELECT * FROM users", (error, data) => {
  // 훗날 요리가 완성되어 주방에서 진동벨을 울리면, 그때 돌아와서 이 서빙(함수)을 실행합니다.
  console.log("요청하신 데이터가 준비되었습니다.", data);
});
console.log("주문을 주방에 넣었고, 저는 곧바로 다음 테이블로 이동합니다!"); // 즉시 실행됨!
```

- 최신 React/NextJS 환경에서 Server Component를 다룰 때 async/await를 사용하는 이유도 내부 V8 엔진에서 비동기 논블로킹으로 처리하는 것을 넘어 일반 작업보다 우선 순위가 높은 Microtask queue로 분류하여 빠르게 처리해 서버의 자원 낭비를 막는다.

## 핵심 용어 상세 해설

### 동기 (Synchronous) vs 비동기 (Asynchronous)

작업의 순서와 시점'을 맞추는 방식의 차이입니다. 동기식은 앞선 작업이 완전히 종료되어야만 다음 줄의 코드를 실행하며(순차적 보장), 비동기식은 작업의 완료를 기다리지 않고 곧바로 다음 코드를 강제 실행합니다.

자바스크립트는 본질적으로 하나의 메인 스레드만 갖습니다. 따라서 데이터베이스를 조회하는 fs.readFileSync(동기)를 사용하면 응답이 올 때까지 전체 서버가 굳어버립니다. 이를 방지하기 위해 작업 완료 후 실행될 함수(콜백)를 예약해두고 다음 로직으로 넘어가는 비동기 방식을 Node.js의 기본 표준으로 채택하고 있습니다.

### 블로킹 (Blocking) vs 논블로킹 (Non-blocking)

'제어권(실행 권한)'을 어떻게 돌려주는지에 대한 운영체제 레벨의 개념입니다. 블로킹은 호출된 함수가 자신의 모든 작업을 마칠 때까지 제어권을 꽉 쥐고 놓지 않는 것이며, 논블로킹은 작업을 백그라운드에 넘긴 직후 즉시 제어권을 돌려주어 호출한 쪽이 다른 일을 할 수 있게 해줍니다.

블로킹은 웨이터가 주방 안까지 들어가서 요리가 끝날 때까지 멍하니 서 있는 것이고, 논블로킹은 주방에 주문서만 찔러 넣고 즉시 홀로 튕겨져 나와 다른 손님을 받는 것입니다. Node.js는 이 '비동기 + 논블로킹'의 조합을 통해 V8 엔진이 단 1밀리초도 쉬지 않고 연산하게 만듭니다.

### 이벤트 루프 (Event Loop)와 이벤트 큐 (Event Queue)

이벤트 루프는 Node.js의 심장으로, 콜백 함수들을 순서대로 메인 스레드에 올려주는 '무한 회전 감시자'입니다. 이벤트 큐는 완료된 작업(콜백)들이 대기하는 '컨베이어 벨트'입니다.

비동기로 던져둔 무거운 I/O 작업들이 OS에서 완료되면, 그 결과(콜백)가 이벤트 큐에 쌓입니다. 이벤트 루프는 홀(콜 스택)이 비어있을 때마다 이 큐를 확인하여 대기 중인 콜백을 순서대로 실행시킵니다. 이 루프가 멈추는 순간(블로킹 발생), 서버의 심장이 멎는 것과 같습니다.

### 스레드 (Thread)와 컨텍스트 스위칭 (Context Switching)

스레드는 프로세스 내에서 실제로 코드를 실행하는 흐름의 최소 단위입니다. 컨텍스트 스위칭은 CPU가 여러 스레드를 동시에 처리하는 것처럼 보이기 위해, 현재 스레드의 상태를 저장하고 다음 스레드의 상태를 메모리에서 불러오는 교체 작업입니다.

Java의 Spring이나 Tomcat 같은 전통적 서버는 접속자마다 스레드를 만듭니다. 1만 명이 접속하면 1만 번의 컨텍스트 스위칭이 초당 수천 번 발생하며, CPU는 실제 연산보다 '자리 바꾸는 작업'에 리소스를 다 써버려 서버가 다운됩니다(Thrashing). Node.js는 싱글 스레드로 이 교체 비용을 '0'으로 수렴시켰습니다.

### I/O 집약적 (I/O Bound) vs CPU 집약적 (CPU Bound)

애플리케이션 성능의 병목(발목을 잡는 요소)이 어디서 발생하는지를 나타냅니다. 네트워크 통신, DB 쿼리, 파일 읽기 등 외부 장치의 응답 대기 시간이 대부분을 차지하면 I/O Bound이며, 복잡한 수학 연산이나 이미지 처리 등 CPU의 연산력 자체가 한계에 달하면 CPU Bound입니다.

Node.js는 논블로킹 I/O를 통해 I/O 집약적인 서비스(채팅 서버, 스트리밍, 실시간 대시보드)에서 세계 최고 수준의 가성비와 성능을 냅니다. 하지만 딥러닝 연산 같은 CPU 집약적 서비스에서는 단일 스레드가 막혀버리므로 절대 메인 서버에서 직접 처리하면 안 됩니다.

### 시스템 콜 (System Call) 및 OS I/O 다중화 (Multiplexing)

사용자 공간에서 도는 프로그램(Node.js)이 하드웨어를 직접 제어할 수 없으므로, 운영체제(커널)의 권한을 빌리기 위해 호출하는 인터페이스입니다.

단일 스레드가 수만 개의 네트워크 연결을 동시에 감당하는 마법의 정체는 epoll(Linux)이나 kqueue(macOS) 같은 커널 레벨의 I/O 다중화 시스템 콜입니다. Node.js는 수만 명의 소켓 상태를 직접 확인하는 대신, OS에게 "누구든 데이터가 도착하면 나한테 알려줘(Notify)"라고 위임해 버림으로써 극단적인 효율을 달성합니다.
