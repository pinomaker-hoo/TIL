# 1강: Node.js의 본질과 운영체제 기반의 런타임 환경 이해

## 개요

NodeJS는 javascript 코드를 OS 위에서 직접 실행할 수 있게 하는 Runtime Environment이다. 기존에는 웹 브라우저 위에서만 작동하는 코드를 하드웨어와 서버라는 영역에서 통제할 수 있게 해주었다.

javascript는 OS의 Memory 접근, 하드디스크의 파일 제어와 같은 시스템 레벨의 작업이 브라우저의 보안 정책으로 인해 불가능하였는 데, NodeJS는 구글 Chrome의 고성능 javascript 엔진 V8을 떼어네고 이를 C++ 프로그램으로 감싸 시스템 레벨의 작업을 가능하게 한다.

서비스는 I/O 집약적 서비스와 CPI 집약적 서비스로 구분이 가능하다.

### I/O 집약적 서비스(I/O Bound)

데이터의 입출력(Input/Output)이 빈번하게 일어나는 서비스를 의미하며 카카오톡, 디스코드와 같은 실시간 채팅이나 넷플릭스와 같은 스트리밍, 데이터베이스의 IO가 여기에 속하는 데 NodeJS는 이러한 요청을 대기 상태 없이 비동기적으로 아주 가볍게 빠르게 넘기며 처리하는 데 이점이 있다.

### CPU 집약적 서비스(CPU Bound)

CPU가 복잡한 수학적 연산을 오랫동안 수행해야하는 작업으로 4K 비디오의 압축, 수백만 건의 금융 데이터 통계 계산, 인공지능 모델 학습등이 포함된다. NodeJS는 한 번에 1가지 작업을 처리하는 특성이 있어, 무거운 계산을 하나 처리하는 동안 다른 요청이 멈추는 Blocking이 발생할 수 있다.

## 용어 정리

### 샌드박스(Sandbox)

외부의 접근이나 영향을 차단하여 보호된 격리 구역을 의미한다.

해킹 사이트의 javascript 코드가 내 컴퓨터의 C드라이브를 포멧하거나 비밀번호를 빼갈 수 없는 이유이며, 브라우저는 보안을 위해 자바스크립트를 철저히 OS에 접근하지 못 하도록 막는 데, NodeJS는 이를 Sandbox에서 꺼내 시스템 제어권을 쥐어준 것이다.

### 런타임 환경(Runtime Environment)

특정 프로그래밍 언어로 작성된 코드가 실행되는 환경

언어는 단순한 텍스트의 불과하지만 이걸 컴퓨터가 알아듣게 번역하고 실행해주는 토대를 Runtime이라고한다. F.E 개발자가 사용하는 window, document 객체는 Browser Runtime이 제공하는 도구이며, B.E 개발자가 사용하는 require('fs')와 같은 파일 시스템 도구는 NodeJS Runtime이 제공하는 도구다.

### API(Application Programming Interface)

프로그램들이 상호작용하기 위한 통신 인터페이스

### 병목 현상 (Bottleneck)

전체 시스템의 성능이 어떤 하나의 느린 요소 때문에 제한받는 현상

도로의 차선이 4차선에서 1차선으로 줄어들 때 차가 막히는 것과 같은 것이며, NodeJS는 싱글 스레드이기에 병목 현상이 매우 치명적입니다. 하나의 무거운 작업이 발생할 경우 다른 요청까지 멈춰버릴 수 있기에 무거운 연산의 경우 Python 서버 등 다른 곳으로 MS하여 병목을 방지하는 처리가 필요합니다.

### 스레드(Thread)

프로세스 내에서 실제로 작업을 수행하는 가장 작은 흐름의 단위

식당에서 일하는 종업원으로 이해하면 좋습니다. Java/C++ 기반의 서버는 손님이 올 때마다 신규 종업원(Thread)를 고용하여 1:1로 응대하게 합니다. 이를 Multi-Thread라고 합니다. NodeJS는 빠르고 기억력이 좋은 종업원(Thread) 한 명이 주문만 재빠르게 받고, 요리는 주방(OS)로 넘기고, 완성되면 알림(Event)를 받고, 다시 손님에게 서빙하는 방식을 쓰는 데 이를 Single-Thread라고 합니다. NodeJS는 이덕분에 수만명의 가벼운 주문(I/O)를 처리하는 데 좋은 성능을 자랑하지만, 종업원이 오래 걸리는 요리를 직접해버리면 다른 주문을 받지 못한다.
