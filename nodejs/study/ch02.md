# 2강. 브라우저를 탈출한 자바스크립트: C++와 V8 엔 진이 만든 런타임의 기적

## 개요

자바스크립트는 High Level Language이며, 이를 CPU가 이해하는 방법, CPU는 0, 1로 이루어진 기계어만 이해합니다. 따라서 javascript 코드를 실행될려면 Engine이라는 통역사가 필요합니다.

원래 CPU는 저전력 설계 기반의 ARM, 고성능 연산 중심의 X86(AMD) 등이 있으며 각자 알아듣는 기계어가 다릅니다. 구글의 V8이나 파이어폭스의 스파이더몽키와 같은 엔진은 우리가 작성한 코드를 해당 컴퓨터의 CPU 아키텍쳐에 맞는 기계어로 실시간 번역하며 전달 합니다.

- 최근 AWS에서는 X86 기반 인스턴스보다 가성비가 좋은 ARM 기반의 Graviton 인스턴스를 제공하는 데, C++, Go와 같은 언어는 이 서버로 옮길 때 타깃 아키텍쳐에 맞게 코드를 다시 빌드해야하지만, NodeJS는 V8 엔진이 현재 아키텍쳐를 감지아호 맞춤형 기계어로 번역해주기에 비즈니스 코드를 수정할 필요 없이 배포가 가능합니다.

## 자바스크립트의 진화: 인터프리터와 컴파일러의 환상적인 혼종, JIT

과거의 javascript는 코드를 한 줄씩 읽어가며 실행하는 인터프리터 방식을 사용하여 속도가 매우 느렸으나 V8 엔진은 빠른 실행 시작과 압도적인 연산 속도를 위해 아래와 같이 조치합니다.

- 먼저 코드를 가벼운 바이트 코드로 빠르게 통역하여 실행을 시작 (Ignition)

- 동시에 반복적으로 사용되는 핵심 코드(Hot Code)를 감지하면 엔진이 이를 백그라운드에서 CPU가 100% 이해하는 기계어로 실시간 번역함 (Turbofan)

- 이를 통해 javascript는 C++에 버금 가는 실행 속도를 가진다.

다른 엔진은 소스코드 > 바이트코드 번역 > 실행의 흐름을 가진다.

```
대규모 트래픽을 받는 서버를 Auto-scaling으로 새로 띄우면 초기 서버가 느리게 동작하는 현상이 있다. V8 엔진이 아직 코드를 바이트코드로만 실행하며 어떤 코드가 Hot Code인지 분석 중이기 때문이다. 트래픽이 누적되면 TurboFan이 즉시 기계어로 컴파일하여 성능이 향상된다. 이를 시스템 설계에선 Server Warm-up이라고 부르며, 서버가 트래픽을 받기 전에 인위적인 요청을 보내 엔진을 미리 최적화 시키는 기법을 사용하기도 한다.
```

## Node.js의 탄생: V8 엔진의 시스템 이식

2009년 라이언 달은 브라우저 내에 있던 V8 엔진을 꺼내 C++ 프로그램 내에 내장 시켰습니다. 즉 NodeJS는 V8 엔진을 심장으로 가진 C++ 어플리케이션이다. 브라우저 환경이 window, document와 같은 UI 도구를 제공했다면, NodeJS 환경은 fs, http 등 OS를 직접 다룰 수 있는 도구를 제공한다.

```
현대의 마이크로서비스 아키텍처(MSA)에서 Node.js는 특히 BFF(Backend For Frontend) 패턴의 핵심 기술로 사용됩니다. 브라우저의 DOM을 다루던 언어적 특성을 살려 클라이언트(React, Vue 등)의 요청을 쉽게 파싱하고, 이식된 운영체제 제어 능력(http, fs)을 통해 백엔드의 여러 마이크로서비스(Java, Python 등)로 비동기 네트워크 요청을 뿌린 뒤, 그 결과를 모아 빠르게 클라이언트에게 반환하는 완벽한 미들웨어 역할을 수행합니다.
```

## Node.js의 3단계 분업 아키텍처

javascript 코드가 OS를 움직이는 과정을 아래의 3과정으로 이루어집니다.

1. javascript 코드 작성 : 개발자가 로직을 작성

2. V8 엔진의 통역 : javascript를 C++이 이해할 수 있는 형태로 통역

3. C++ 코어 및 바인딩(Bindings) : 통역된 지시를 받아 OS에 직접 명령을 내림.

```
실무에서 API 응답 속도가 느려질 때, 엔지니어는 이것이 V8의 영역(CPU를 많이 쓰는 로직)인지, C++ 바인딩의 영역(네트워크나 파일 읽기 대기)인지 분간해야 합니다. 만약 이미지 리사이징이나 대규모 암호화처럼 철저하게 CPU를 혹사시키는 작업이라면 자바스크립트(V8)로 처리하는 것은 비효율적입니다. 이런 경우, 현업에서는 해당 부분만 C++ Addon으로 작성하여 직접 연결하거나, Rust/Go 같은 언어로 별도의 워커(Worker)를 띄워 시스템 부하를 분산시키는 설계 결정을 내립니다.
```

## 운영체제 관점에서의 실행과 관리

NodeJS를 설치하고 환경변수(PATH)를 설정하여 node라는 명령어만으로 터미널에서 조작할 수 있게하는 것은 OS의 PATH에 NodeJS 실행 파일의 위치가 기록되어있기 때문이다.

NodeJS가 실행되면 OS는 Process ID를 부여하고 메모리를 할당하며, 이를 통해 OS는 NodeJS가 자원을 얼마나 쓰는 지 감시/통제가 가능합니다.

```
실무에서 서버 업데이트를 할 때 사용자가 접속 끊김을 느끼지 못하게 하는 '무중단 배포(Zero-downtime Deployment)'의 핵심이 바로 이 PID 관리입니다. PM2 같은 프로세스 매니저는 운영체제에 여러 개의 Node.js 프로세스(각기 다른 PID)를 띄웁니다. 업데이트 시 기존 PID로 들어오는 요청은 마저 처리하게 두고(Graceful Shutdown), 새로운 PID를 가진 프로세스를 띄워 트래픽을 전환합니다. 또한, 싱글 스레드인 Node.js가 8코어, 16코어 CPU를 100% 활용하기 위해 코어 개수만큼 PID를 복제하는 '클러스터링(Clustering)' 기법의 근간이 됩니다.
```

## 용어 정리

### CPU 아키텍쳐(ARM vs x86)

CPU가 명령어를 처리하는 물리적인 설계 방식으로, 우리가 짠 코드가 그 아키텍쳐에 맞게 엔진이 알아서 다른 0과 1의 조합으로 기계어를 생성해냄.

특히 Docker를 사용해 Image를 빌드할 때 개발자의 Apple Silicon 맥북에서 빌드한 이미지가 X86 기반의 AWS EC2에서 돌아가지 않는 "exec format error"의 근본적인 이유다. -> 졸라 많이 겪음

### 컴파일(Compile)

사람이 읽기 쉬운 High Level Language를 컴퓨터가 읽는 기계어로 일괄 번역하는 과정을 의미하며, C++ 같은 언어는 실행 전에 번역을 미리 끝내두어 속도가 매우 빠른 AOT(Ahead-Of-Time) 방식을 사용하지만 V8 엔진은 JIT(Just-In-Time) 컴파일을 사용하여 실행과 동시에 번역과 최적화를 병행하여 유연성과 속도를 동시에 챙김.

Vercel, Lambda와 같은 Serverless 환경에선 함수가 처음 호출될 때 Cold Start 지연 현상을 이해하고 완화하는 아키텍쳐를 설계하는 데 도움을 줌

### 바이트코드(Bytecode), JIT 컴파일

바이트코드는 기계어 직접 단계의 효율적인 중간 언어이며, JIT 컴파일은 프로그램 실행 중에 실시간으로 기계어로 번역하는 기술을 의미한다. V8 엔진은 메모리를 아끼기 위해 기본적으로 바이트코드를 사용하지만, 성능이 필요한 순간에는 기계어로 변환하여 사용하는 전략을 취하는 데 이는 NodeJS의 강력한 퍼포먼스의 기반이다.

javascript는 객체의 구조가 동적으로 변할 수 있는 데, 구조가 계속 변하면 V8의 JIT 컴파일러(TurboFan)이 기계어로 최적화해둔 코드를 다시 폐기(Deoptimization)하여 성능이 급락하는 데, 실무에선 Typescript를 사용해 객체 구조를 고정시키거나, Hidden Class가 깨지지 않게 코딩하여 엔진 최적화를 신경 써야함.

### C++ 바인딩

javascript의 명령을 C++의 기능과 연결해주는 가교를 의미한다. javascript는 파일 시스템, 네트워크 시스템에 접근할 수 있는 능력이 없지만, C++ 바인딩 덕분에 javascript에서 특정 함수를 호출하면, C++ 코드가 연결되어 OS의 시스템에 접근할 수 있다.

Database와 통신하는 드라이버 라이브러리 MySQL, pg나 암호화 라이브러리 bcrypt와 같이 I/O 성능이나 연산이 필요한 패키지는 수수 JS가 아닌 C++ 바인딩(이하 Node API)를 통해 작성되어있어 성능이 극대화됨.

### libuv와 Thread Pool

libuv는 비동기 I/O를 작업을 처리하는 C 라이브러리이며, 작업을 나누어 받는 일꾼 대기실이다. NodeJS는 기본적으로 4개의 Thread를 Thread Pool에 할당한다.(UV_THREADPOOL_SIZE=4)

NodeJS가 Single Thread이지만 강력한 이유는 시간이 오래 걸리는 파일 작업등을 libuv의 Thread pool에 있는 Thread에게 할당하고 다른 요청을 처리하기 때문이다.

서버에서 DB 쿼리나 파읽 읽기가 동시에 수백개씩 발생한다면, 4개의 기본 Thread로는 병목이 발생하여 서버 전체가 멈출 수 있기에, 서버의 코어수와 트래픽을 계산하여 OS의 환경변수 UV_THREADPOOL_SIZE를 튜닝해야합니다.

### Process, PID

메모리에 올라가 실행 중인 상태의 프로그램과 그 프로그램을 식별하는 고유번호다. 서버를 운영할 때 PID를 기반으로 프로세스를 찾아 종료하거나 자원 사용량을 제한하는 등의 관리가 가능하며, Datadog, Prometheus와 같은 모니터링 시스템을 구축할 때 수많은 인스턴스 중에서 메모리 누수가 발생한 특정 PID를 추적하여 컨테이너를 재시작시키는 자동화된 복구 Self-healing 시스템을 설계하는 기준이 됩니다.
